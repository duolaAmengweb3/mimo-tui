# 13 · 发布后运营

> 目标：发布后 1–3 个月维护好产品 + 持续推广 + 收集反馈
> 估时：持续（每周 5–10 小时）
> 依赖：12 发布完成

## 13.1 第一周

### 监控
- [ ] 每天看 GitHub Issues + 标 label
- [ ] 每天看 Discord / 微信群
- [ ] 每天看社交媒体 mention
- [ ] 监控 npm 下载量 / GitHub stars 增长

### 响应
- [ ] Issue 响应中位数 < 24h
- [ ] 严重 bug 24h 内 hotfix
- [ ] 一般 bug 1 周内 fix

### 文档
- [ ] 把首周高频问题加到 FAQ
- [ ] 修复文档里的错别字 / 死链
- [ ] 补充缺失的截图

## 13.2 第一个月

### 版本节奏
- [ ] 第 2 周：v1.0.1（首日 P1 bug 修复）
- [ ] 第 3 周：v1.0.2（社区反馈中的高频改进）
- [ ] 第 4 周：v1.1.0（第一个 minor，加几个用户最要的小功能）

### 内容输出
- [ ] 每周 1 篇技术博客（推到知乎 + 公众号）
- [ ] 主题候选：
  - "如何写一个 Anthropic 协议的 Rust client"
  - "ratatui 实战：5 个让 TUI 不卡的技巧"
  - "为什么我们不 fork Claude Code"
  - "MiMo 模型的 thinking 字段怎么用"
  - "从 0 做一个全平台 CLI 分发"

### 社区
- [ ] Discord / 微信群人数 > 100
- [ ] 至少 1 个外部贡献者 PR 合并
- [ ] 启动 Skills 市场（鼓励社区贡献 skill）

### 推广延伸
- [ ] 上 awesome-cli-coding-agents
- [ ] 上 awesome-rust
- [ ] 上 awesome-mimo-agent（如果存在）
- [ ] 找 3 个 KOL 试用 + 推荐

## 13.3 第二、第三个月

### 产品迭代
- [ ] v1.2 / v1.3 按用户反馈持续迭代
- [ ] 性能优化
- [ ] 更多内置 skills

### 数据复盘
- [ ] 月度数据：stars / 下载 / DAU / issue 关闭率
- [ ] 用户画像分析
- [ ] 高频任务 / 失败任务统计（如有 telemetry）

### 生态扩展
- [ ] 推动 1 个公司 / 团队作为标杆用户
- [ ] 跟小米官方建立联络（推荐 / 合作 / 收编可能）
- [ ] 考虑矩阵化：启动 Hy3-TUI / Kimi-TUI（用同架构）

### 国际化
- [ ] 推到海外社区（Reddit / HackerNews / Dev.to）
- [ ] 英文文档进一步精修
- [ ] 英文版 demo 视频

## 13.4 长期机制

### 治理
- [ ] 设 maintainer 团队（如有合作者）
- [ ] RFC 流程（重大功能讨论）
- [ ] roadmap 公开

### 沟通
- [ ] Discord 每周 office hour
- [ ] 月度社区 newsletter（可选 Buttondown 免费 tier）

### 反爬 / 反滥用
- [ ] 监控异常使用模式
- [ ] 防止竞品直接 fork 然后改皮（用 MIT 但加 Attribution）

## 13.5 失败 / 退路机制

- [ ] 3 个月 stars < 500 → 重新评估推广策略
- [ ] 6 个月 DAU < 100 → 考虑深度合作或开源给基金会
- [ ] 永远不删仓库（哪怕维护停了也保留）

## 验收（发布 3 个月后）

- [ ] Stars > 3000
- [ ] npm 月下载 > 5000
- [ ] 至少 5 个外部 PR 合并
- [ ] 至少 10 个社区 skills
- [ ] Discord / 微信群 > 100 人
- [ ] 有第三方教程 / 评测 / 视频
- [ ] 跟小米官方有联络
