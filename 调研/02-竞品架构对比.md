# 竞品架构对比

> 时间：2026-05-19
> 目的：搞清楚跟 MiMo TUI 抢用户的所有项目长什么样、做到什么程度、留下什么空白

## 一、直接竞品（MiMo 专属工具）

### 1. XIAOMI-MiMo-code（Gurabit77）⭐ 唯一直接竞品

**项目地址**：https://github.com/Gurabit77/XIAOMI-MiMo-code

| 维度 | 数据 |
|---|---|
| Stars | **25** |
| Forks | 6 |
| Commits | 511 |
| 语言 | TypeScript 99.8% |
| 最近活跃 | 持续更新中 |

**核心定位**：小米 MiMo 模型专属 AI 编程助手，终端命令 `mimo`

**实现路径**：
- **基于 CCB（Claude Code Best）逆向工程改造**
- CCB 是 Anthropic Claude Code CLI 的开源逆向还原项目
- MiMo Code 在 CCB 基础上做了 MiMo 适配

**与 Claude Code 隔离**：
- 配置目录：`~/.mimo/` vs `~/.claude/`
- 缓存路径独立
- Unix Domain Socket / Windows 命名管道独立
- 不共享 Keychain
- 模型列表只显示 MiMo

**安装**：克隆仓库 → `bun install` → `npm link`，需 Node 18+ + Bun 1.3+

**已解决的痛点**：
- 系统提示词避免 "You are Claude" 的自我认知混乱
- 配置目录冲突
- 模型列表混淆

**关键弱点**：
1. **25 stars / 6 forks** —— 远未成气候
2. **法律灰色**：基于逆向 CCB（Claude Code 逆向项目），Anthropic 随时可发 DMCA
3. **TypeScript + Bun** —— 安装步骤多，对非开发者不友好
4. **没有自己的差异化**——只是"换皮 Claude Code"

**我们的差异化空间**：
- 走**原创架构**（不依赖逆向）→ 法律安全
- 走 **Rust + 单二进制** → 分发更轻
- 做 **MiMo 深度优化**（thinking 可视化、缓存命中、套餐用量）→ 不只是套壳

### 2. mimo2codex（7as0nch）

**项目地址**：https://github.com/7as0nch/mimo2codex

**性质**：**不是 TUI，是 Codex CLI 的本地代理 shim**

**做什么**：
- 把 Codex 的 Responses API 实时翻译成 Chat Completions
- 让 Codex CLI 能调 MiMo / DeepSeek / Qwen / GLM / Kimi / Ollama 等
- 解决"Codex 新版硬错误于 MiMo 老 API"问题

**技术栈**：TypeScript 85% + Python 6%

**已知限制**：
- `/hatch` 命令硬编码调用 OpenAI 图像生成，MiMo 无此 endpoint
- 仅 mimo-v2.5 和 mimo-v2-omni 支持视觉

**对我们的启示**：
- 这是"间接竞品"——用户如果接受"Codex CLI + 代理"方案就不会用我们
- 但代理方案有切换成本和稳定性问题，**原生 TUI 体验更好**
- 它的存在反而**证明 MiMo 用户有强烈需求**

### 3. CC Switch（farion1231）

**项目地址**：https://github.com/farion1231/cc-switch

| 维度 | 数据 |
|---|---|
| Stars | **75.1k** ⚠️ 数据存疑（可能为 7.5k） |
| 语言 | Rust 58.6% + TypeScript 39.4% + Tauri 2 |
| 最近发布 | v3.15.0（2026-05-16） |

**性质**：不是 TUI，是 **5 个 CLI 工具的统一管理器**（Claude Code / Codex / Gemini CLI / OpenCode / OpenClaw）

**主要功能**：
- 一键切换不同 provider（50+ 预设）
- 系统托盘快速切换
- 统一 MCP / 技能管理
- 本地代理 + 自动故障转移
- 云同步（Dropbox / OneDrive / iCloud）
- 使用追踪与成本统计
- 会话管理器

**对我们的启示**：
- 不是直接竞品，是**潜在合作伙伴 / 互补工具**
- 它解决了"我有多个 CLI 怎么管理"的问题
- MiMo TUI 应当**主动被 CC Switch 收录**

## 二、行业范式参考（学这两家的成功路径）

### A. DeepSeek-TUI（Hmbown）—— 单模型 TUI 范式标杆

**项目地址**：https://github.com/Hmbown/DeepSeek-TUI

| 维度 | 数据 |
|---|---|
| Stars | **32.2k** |
| Forks | 2.7k |
| 语言 | Rust 95% |
| 最近版本 | v0.8.39（2026-05-17）|
| 发布次数 | 95 个 release |

**架构亮点**：
- `crates/cli`：命令行入口
- `crates/tui`：TUI 运行时（ratatui）
- `web/`：可选 Web 界面
- `integrations/`：飞书桥接等
- `deploy/`：腾讯 Lighthouse 部署脚本

**功能模块**：
- **三种模式**：Plan（只读探索）/ Agent（交互+审批）/ YOLO（自动批准）
- **推理流式输出**：实时展示 DeepSeek thinking
- **类型化工具注册表**
- **LSP 诊断集成**（rust-analyzer / pyright 等）
- **子 agent 系统**：最多 10-20 个并发
- **会话管理**：保存 / 恢复 / 分叉
- **工作区沙箱**：macOS Seatbelt / Linux Landlock / Windows Job Objects
- **14+ 预置 skills**
- **MCP 服务器集成**
- **Native REPL 持久化**

**安装多渠道**：npm / cargo / brew / 直接下载 / Docker

**差异化卖点**：
- `--model auto` 智能选模型
- 前缀缓存可视化
- 1M 上下文 + 自动压缩

**对 MiMo TUI 的直接借鉴**：
- 选 **Rust + ratatui** 路线（成熟、性能好、单二进制易分发）
- 复制 **三种模式**（Plan / Agent / Auto）
- 复制 **思考流可视化**（用 MiMo 的 thinking 字段）
- 复制 **沙箱设计**（macOS/Linux/Windows 平台原生隔离）
- 复制 **MCP + Skills 系统**
- 多渠道分发（npm / cargo / brew / docker）

### B. OpenCode（SST）—— 多模型 OSS 范式标杆

**项目地址**：https://github.com/sst/opencode

| 维度 | 数据 |
|---|---|
| Stars | **162k** |
| Forks | 19.1k |
| 语言 | TypeScript 65% |
| 最近版本 | v1.15.5（2026-05-18） |

**架构**：
- `packages/`：核心包
- `sdks/vscode`：VS Code SDK
- `infra/`：基础设施
- `nix/` `perf/`：包管理 / 性能

**两种 agent**：
- **build**：完全访问权限做开发
- **plan**：只读分析和探索
- **通用子 agent**：复杂搜索和多步任务

**支持 75+ provider**（不锁模型）

**安装多渠道**：一键脚本 / npm / brew / scoop / pacman / Nix / 桌面 App

**对 MiMo TUI 的启示**：
- 多模型 fallback 是关键卖点（不只是 MiMo，可以无缝切到其他模型）
- 单仓库 / 多包架构利于扩展
- 桌面 App + CLI 并行分发

## 三、竞品 vs 我们的差异化总结

| 项 | XIAOMI-MiMo-code | mimo2codex | CC Switch | **MiMo-TUI（我们）** |
|---|---|---|---|---|
| 性质 | 逆向 CCB | Codex 代理 | 多 CLI 管理器 | 原创专属 TUI |
| 语言 | TypeScript | TypeScript | Rust + TS | **Rust（独立分发）** |
| Stars | 25 | 未查 | 75K | 0（待建） |
| 法律风险 | 高（逆向） | 中（代理）| 低 | **低（原创）** |
| MiMo 深度 | 改 prompt | 仅 API 翻译 | 仅配置切换 | **深度（thinking / 缓存 / 套餐）** |
| 安装易度 | 中（要 Bun）| 中 | 高 | **高（单二进制 / npm / brew）** |
| 矩阵化潜力 | 低 | 低 | - | **高（同架构复制到 Hy3 / GLM）** |

---

## 四、结论

**MiMo 专属 TUI 这个赛道仍然空着**：
- 唯一直接竞品 25 stars 且基于逆向（不靠谱）
- 间接方案（代理、管理器）体验割裂
- 没有"DeepSeek-TUI for MiMo"级别的原生 TUI

**我们能赢的路径**：
1. 用 Rust + 单二进制建立分发优势
2. 用原创架构避开法律风险
3. 用 MiMo 深度优化（thinking / 缓存 / 套餐 / 1M 上下文）建立体验优势
4. 用矩阵化思维设计架构（未来同代码库复制给 Hy3 / GLM / Kimi）
